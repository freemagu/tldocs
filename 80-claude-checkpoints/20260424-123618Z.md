## Decisions made (and why)

- **18 features (not 21)**, dropping `atr_anchor_abs`, `previous_breach_market_label_rejected`,
  `previous_breach_first_cross_back_seconds_log` ‚Äî user explicitly asked for the trim
  before first MVP run. The last two have first-breach NaN sparsity; `atr_anchor_abs` is
  a raw-scale feature already implicit in ATR-normalised neighbours.

- **Fixed œÑ = 0.5 for headline F1**; best-train-fold œÑ reported but explicitly labelled
  "secondary diagnostic" in both code (`_full_metrics`) and the output report.
  Per user: "Do not make 'best F1 threshold chosen on the train fold' the main reported
  operating point."

- **Prior-rate baseline uses train base rate `mean(y_train)` for first-breach rows**,
  NOT 0.5. `prior_rate_baseline(test_df, y_train)` hard-codes this and is unit-tested
  (`test_prior_rate_baseline_uses_train_base_rate_for_first_breach`).

- **Grouped forward split with ties broken on `level_id` ascending.** Deterministic split,
  unit-tested at both tie-free and tied-timestamp inputs. Every test-side level was confirmed
  strictly after every train-side level.

- **Single shared feature matrix for all four targets** (not per-target cutoffs). The
  leakage rule "every scalar strictly at or before `breach_ts`" applies to the tightest
  horizon (15s), rules out all post-breach data, and is simplest. Worth small signal loss
  on 180s for zero-bug leakage guarantee.

- **Four separate binary classifiers (LR + GBM per target), not shared / ordinal.**
  Monotonic consistency can be enforced post-hoc; per-bucket thresholds decoupled;
  failure isolation; trivially parallel. Revisit if LR and GBM coefficients converge on
  identical shape ‚Äî they don't (different top-3 across targets), so this stays for MVP.

- **`signed=True` in `aggregate_feature_importance` for LR** (reports `sign_flips`);
  `signed=False` for GBM (unsigned importances, sign-flips column omitted).

- **`same_bar_event` is not a feature.** Constant 100 % on the re-armed dataset.
  A test (`test_same_bar_event_is_not_in_feature_schema`) asserts its absence from
  `MVP_FEATURE_COLUMNS`.

- **Ticks with `ts ‚â• breach_ts` are filtered inside the microstructure function**
  (not relying on caller pre-filtering). Defensive "leakage guard" test confirms the
  output is identical with/without post-breach poisoning.

- **At-breach tick price (first tick of `ticks_after`) IS admissible** as
  `breach_magnitude_at_tick_atr`. The breach tick is known AT breach_ts (it's the event
  that defines breach_ts), not after.

- **Reconstruct `total_prior_rejections_on_level` from `rate √ó breaches` in
  `load_features`** rather than re-running extraction to add a passthrough column. The
  rate is a small-integer ratio (‚â§ 40 breaches max) so `round(rate * breaches)` is exact.

- **Installed `scikit-learn==1.7.2` into the shared venv.** No project-level
  requirements.txt / pyproject deps list to update; documented in the final report so
  user knows the venv mutated.

## Rejected approaches (and why)

- **Per-target cutoffs (different feature sets per horizon).** Rejected at design time ‚Äî
  simpler bookkeeping, same-for-all-targets feature matrix, minor signal-loss trade-off
  accepted. User agreed with the simplification implicitly (didn't flag it in the 7-item
  revision list).

- **Predict all four targets with a single multi-output / ordinal model.** Rejected for
  MVP; revisit only if LR/GBM show strong convergence. They don't, so four separate.

- **Hardcode 0.5 for first-breach rows in prior-rate baseline.** User explicitly rejected
  this in revision list: use train base rate instead.

- **Use the last pre-breach tick price as `breach_magnitude_at_tick_atr`.** Would give
  "close to the line" signal rather than actual breach depth. Rejected ‚Äî use the
  first tick of `ticks_after` which IS the first crossing tick; this is admissible
  under the leakage rule (at-breach, not post-breach).

- **Rename the old pre-v2 Phase 3 files to `_legacy` / `_prev2`.** Considered, chose
  instead to overwrite the CLI (`swing_levels_phase3.py`) and leave `features.py`
  + `test_swing_features.py` as orphaned dead code. Git preserves history. User can
  delete those manually if desired.

- **Add `total_prior_rejections_on_level` to features.csv passthrough and re-run
  extraction.** 45-min wasted cost for a column add. Chose inline reconstruction in
  `load_features` instead; zero re-extraction. (Should be fixed in the pipeline for
  future runs, but deferred.)

- **`EXTRA_COLUMNS` tuple separate from `IDENTITY_COLUMNS` in the pipeline.** Considered
  after discovering the reconstruction was needed. Deferred ‚Äî the reconstruction is
  reliable and there's no other extra column to carry today.

- **Change `prior_rate_baseline` signature to take the rate column directly instead of
  raw counts.** Would break the existing unit test which constructs
  `total_prior_rejections_on_level` directly. Rejected for signature-stability reasons.

- **`pre_large_tick_count_300s`, `pre_vol_atr_60s`, `regime_vol_4h_atr`,
  `pre_velocity_atr_5s`, `bar_so_far_direction`, `prior_bar_close_position`.** All

## Surprises / gotchas

- **`market_candle` PK is `(symbol, market_type, timeframe, open_time)` ‚Äî WITHOUT
  `exchange`.** My initial integration test's `ON CONFLICT` specified all 5 columns
  including exchange and failed with `InvalidColumnReference: there is no unique or
  exclusion constraint matching the ON CONFLICT specification`. Also `source` is NOT
  NULL and must be set (I used `'test'`). Fix at `tests/integration/test_phase3_pipeline.py`
  around the `_seed_candles` helper.

- **Features.csv was missing `total_prior_rejections_on_level`.** The pipeline passes
  `IDENTITY_COLUMNS` + features + labels; `total_prior_rejections_on_level` is a raw
  Phase 2 count used only by the training baseline ‚Äî it's neither an identity column
  nor a feature, so it was dropped. The training script reconstructs it from
  `rate √ó breaches`. Recognised too late to fix the pipeline without a re-extraction.

- **The `_min_phase2_row()` test helper missed `level_price`.** Two tests failed on
  `KeyError: 'level_price'` until I added it. The composer reads `level_price` from
  the phase2_row dict, which is the sensible design for pipeline callers (the real
  events DataFrame has `level_price` as a column).

- **Phase 3 PR-AUC is monotonic-increasing with delay horizon** (0.68 ‚Üí 0.71 ‚Üí 0.75
  ‚Üí 0.82). Easier to predict "safe to delay 3 min" than "safe to delay 15 sec" ‚Äî
  consistent with positive-rate rising and features carrying more signal for bigger
  windows.

- **`prior_rejection_rate_on_level` has a consistent NEGATIVE coefficient** across all
  four targets (sign_flips = 0). Counterintuitive ‚Äî reads as "levels that historically
  rejected a lot are LESS likely to reject now". Stable signal, worth naming not
  correcting.

- **`pre_velocity_atr_15s` has positive coefficient** across all four targets. Initially
  confusing ‚Äî positive velocity means price is moving toward the breach direction. The
  consistent positive sign suggests "momentum-driven breaches are more reversible,
  gap/air breaches are sticky". Plausible reading but worth a sanity check later.

- **100 % `same_bar_event=True` means it is NOT a usable feature** on the re-armed
  dataset. Explicitly excluded by design + sentinel test.

- **TickLoader path still uses `get_archive_config()["root_dir"]`** ‚Äî the bug from Phase 2
  (where manually constructing `/db/data01/tick_archive/tick_trade_raw/bybit` doubled
  the path) would recur if someone tries to skip the config helper. Documented in the
  Phase 2 checkpoint; avoided here by consistent use of `get_archive_config()`.

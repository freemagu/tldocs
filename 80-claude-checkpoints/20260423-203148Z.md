## Decisions made (and why)

- **3600 s observation window** (not 1800). One tick load per event covers Layer B + future re-labelling; I/O cost is negligible; 1800 s was too short for Layer A structural semantics.
- **Per-bucket measurements at `{15,30,60,180,300,600}s`** (ATR-only, PRICE + PCT dropped for buckets; full-window extremes keep PRICE for debugging). PCT is reconstructable from ATR × atr_anchor / level_price; dropping saves ~15 redundant fields.
- **Strict recovery (`first_cross_back_seconds`, `k_recovery_atr=0.1`) drives all labels**; permissive `first_touch_back_seconds` kept as diagnostic only. Kiss-and-continue is not a recovery.
- **Layer A = {rejected, not_rejected}** (2-class). Dropped `held` → `not_rejected`: `held` overstated the claim ("level structurally held"); `not_rejected` honestly names "we did not observe a strict cross-back".
- **Layer B booleans, not mutually-exclusive categorical.** Nested by construction (`safe_15s → safe_30s → safe_60s → safe_180s`). `smallest_safe_bucket` stays as a reporting-only convenience field.
- **`market_label` NULL on tick-gap**, with a diagnostic `market_label_bar_fallback` sidecar. Do NOT conflate bar-fallback with tick-derived in ML training.
- **`execution_label='unknown'` on tick-gap**, distinct from `unsafe_to_delay`. Two different semantics must never merge.
- **Re-arm = fully-inside bar** (not close-inside, not K×ATR). ATR-free, parameterless, clean "episode" semantics: after a breach, price must step clear of the level before the next breach counts.
- **Phase 1 NOT modified.** Multi-breach enumeration lives entirely in Phase 2's new `breach_enumerate.py` module; Phase 1's `breach_detect.py` (first-breach single-event) preserved untouched.
- **Adverse excursion NEVER a Layer B veto.** LevelGuard only manages soft stops; a separate hard stop caps catastrophic risk. Per-bucket adverse is recorded in raw for downstream risk-budget reasoning but must not gate labels.

## Rejected approaches (and why)

- **Modify Phase 1's `detect_breaches` for multi-breach.** Rejected — user constraint "do not redesign Phase 1". Added `breach_enumerate.py` in Phase 2 instead; Phase 1 output stays byte-identical.
- **K × ATR re-arm threshold.** Rejected — requires plumbing ATR into the enumerator (which currently takes only Pivot + Candle) and adds an arbitrary K parameter. Fully-inside-bar is parameterless and ATR-free.
- **Cluster-based filter (first breach per 24h window).** Rejected — loses legitimate fast-rejection patterns inside a cluster. Re-arm preserves those if price pulled back in between.
- **Level invalidation (X ATR past = dead level).** Rejected — larger design change, crossed the "do not redesign Phase 1" line, and re-arm achieves most of the same effect with less machinery.
- **Minimum gap between consecutive breaches.** Rejected — loses fast genuine SFP patterns.
- **Fixed per-level cap (first N events only).** Rejected — blunt, arbitrary cutoff, biases against legitimately-revisited levels.
- **PCT form of excursions.** Rejected — redundant with ATR given level_price + atr_anchor are in the row. Reconstructable on demand.
- **Collapse Layer A to binary {rejected, not_rejected} via magnitude threshold.** The v1 design had `accepted` vs `ambiguous` split by `k_accept_atr ≥ 1.0`. Rejected after pressure-test — defining `accepted` by a continuation magnitude conflates "breach held structurally" with "breach extended". Two-class is honest.
- **`unsafe_to_delay` for tick-gap events.** Rejected — conflates "no data" with "data showed no recovery". Use explicit `unknown` instead.
- **Option 2 re-arm (close-inside, `bar.close < L`).** Considered during pressure-test analysis. Settled on Option 1 (fully-inside, `bar.high < L`) as cleaner "episode" semantics — a bar that wicks above and closes inside still engaged with the level and isn't a clean pullback.
- **Keeping the permissive 9,444-event population.** Diagnosed and rejected: 80% of events were breach_idx>10 oscillation chatter; median inter-breach gap was 30 min; chasing classifier signal against that population would model "is this chatter?" rather than the soft-stop decision.
- **Running the full 2-hour BTC pipeline before approving re-arm.** User explicitly gated this; diagnostic was run first, proved chatter hypothesis, then re-arm was implemented, re-diagnosed, now awaiting approval again.

## Surprises / gotchas

- **TickLoader `archive_root` semantics:** it internally appends `tick_trade_raw/bybit/{symbol}`. Initialising with `"/db/data01/tick_archive/tick_trade_raw/bybit"` silently returns `data=None` on every load because the constructed path doubles up. Must use `archive_cfg = get_archive_config()` from `tradelens.tick_archive.config` and pass `archive_cfg["root_dir"]`. Cost: one debug cycle during smoke test.
- **Permissive enumerator produces 9,444 events from 191 levels** — far more than the user's "5-20 per level" expectation. Root cause: bar-level state machine with inclusive recovery (`bar.low <= L`) produces an event on every oscillation, with no hysteresis. Needed re-arm rule to fix.
- **All AFTER events are `same_bar_event=True`** after re-arm. Initially looked suspicious but it's correct: for swing-high levels, a fresh organic crossing from the inside naturally produces a bar with `low <= L` (came from below) and `high > L` (crossed above). The 22% of BEFORE events that were non-same-bar were exactly the stale continuation artefacts.
- **Pivot dataclass has `index` field** that refers to position in the candle series used during Phase 1 detection. My enumerator does NOT use this index (walks all candles with `bar.open_time < confirmed_at` continuation), so CSV-loaded Pivots can use any index value — irrelevant.
- **Minor test construction trap:** INSIDE-state + same-bar breach+recovery case is easy; BREACHED-state + same-bar is the ambiguous Case D. First attempt at `test_single_breach_no_recovery` used bars like `(99.5, 101, 99.5, 100.5)` which inadvertently triggered same-bar recovery (low=99.5 <= 100). Required gap-up construction `(101, 102, 100.5, 101.5)` to get a no-same-bar breach.
- **Test-plan skill:** this session triggered the `/test-plan` skill multiple times. It was followed: every production change here went through a test plan and test-first ordering. All 79 unit tests currently green.

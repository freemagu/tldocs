## Decisions made (and why)

- **Partition the backlog before attempting any fixes.** 366 findings span
  typo-level cleanups to 6,700-LOC component splits; no single runner
  shape works for all of them. Four tiers make each finding's disposition
  explicit before any code moves.
- **T1 criteria are deliberately strict.** Forced to T2 if: signature
  changes with external callers, DB schema touched, grep shows non-zero
  callers for a "dead" function, or the file is money-moving code
  (`level_guard_daemon.py`, `api/trades.py`, `api/open_orders.py`,
  `api/suspend.py` — additive-only changes OK in these).
- **Pilot triage first, then user review, then continue.** The user can
  sanity-check my T1/T2/T3 calls on a manageable 77-item slice before I
  classify the remaining 289.
- **Serial batches per file, not parallel worktrees.** Agents writing the
  same file in separate worktrees produce unmergeable diffs; per-file
  batching amortises test setup.
- **Regression test before fix, per existing `/test-plan` policy.** The
  audit exists partly because tests were skipped — dropping the gate here
  would be the exact antipattern the tracker was meant to prevent.
- **Stop aggressively, not gracefully.** First test regression or
  unexpected caller halts the run and hands back to user. Better to stop
  at 3 commits with clear state than push through 20 commits with a
  latent bug.

## Rejected approaches (and why)

- **"One agent handles the whole tracker in one long run."** Rejected —
  336-finding context inevitably drifts; subtle misapplications compound;
  impossible to bisect which commit broke what. Per-batch agents with
  fresh context each time are the right grain.
- **"Parallel fixes in N worktrees."** Rejected — high merge conflict
  rate on same-file findings; extra coordination overhead for little
  speedup when most of the wall-clock is the pre/post test gates anyway.
- **"Skip tests to go faster."** Rejected — testing policy is
  load-bearing for the audit premise. The tracker exists because people
  shipped without tests. Short-circuiting the gate reproduces the problem.
- **"No triage — just pick T1 candidates as we go."** Rejected — too
  easy to misclassify under "just fix it" pressure; explicit tier
  assignment makes the decision reviewable.
- **"Full-tracker triage first, then start."** Considered, not chosen —
  user preferred piloting on chunks 1-2 to sanity-check the rubric before
  committing to 289 more classifications.
- **Starting Batch 1 before checkpointing.** User asked if it was worth
  pushing to "first batch committed" before the checkpoint. Answered no —
  current state is a cleaner cut point (no in-flight edits) than mid-batch.

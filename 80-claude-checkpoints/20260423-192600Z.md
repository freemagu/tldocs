- **`tradelens_test` market_candle table** — doesn't exist in `setup_database_pg.py` because production creates it on-demand via `CandleStorePG.ensure_schema()`. I inlined the DDL into `setup_test_db.py` because the real `CandleStorePG` module can't be imported (depends on deleted `tradelens.mdsync.*`). Documented inline.
- **Candle reader uses `config.postgresql` (separate section from `config.database`).** The test autouse fixture had to swap BOTH to avoid the candle reader accidentally hitting production. Earlier session cleanup (commit `9dbf0d8d`).
- **FP precision at 0.01 tolerance boundary** — `abs(99.99 - 100.0) = 0.010000000000005116` in float, not exactly 0.01. Fixed in AUD-0045 by computing sum via Decimal. Worth remembering for any future money-adjacent float math.
- **`order_leg_event.event_type` NOT NULL** — not surfaced in schema summaries; discovered when writing integration tests. Seed helper defaults to `'amend'`.
- **`order_leg_live` has NOT NULL `updated_at` but NO `created_at`** — unlike most other tables. Inconsistent schema.
- **AUD-0074 test leaked one trade_journal row** in tradelens_test during iteration before I refactored it to a static source check. I cleaned manually via psql: `DELETE FROM trade_leg_map WHERE trade_id IN (SELECT trade_id FROM trade_journal WHERE symbol='BTCUSDT'); DELETE FROM order_leg_hist WHERE symbol='BTCUSDT'; DELETE FROM trade_journal WHERE symbol='BTCUSDT';`. This is the leak that triggered the current open question.
- **Psycopg2 connection/cursor attrs are read-only** (C objects). Any "capture execute args" approach via monkeypatching these won't work; use `inspect.getsource()` source-level checks or `psycopg2.extras.LoggingCursor` instead.

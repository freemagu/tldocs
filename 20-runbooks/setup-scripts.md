# Setup Scripts — Operator Runbook

Single-page runbook for the scripts in `bin/setup/`. Covers the order to run
them for a fresh install, what each one does, when re-running is safe, and
which legacy artifacts in this directory no longer apply post-Sybase.

Resolves audit finding **AUD-0362** ("no README / runbook for 14 setup
scripts").

## Audience

Operators bringing up a new TradeLens environment, the test database, or
running a one-off backfill / migration. If you're only writing application
code, you do not need most of this — `migrate.py up` after a `git pull` is
enough.

## Prerequisites

| Requirement | How to verify |
|---|---|
| `TSHOME=/app/syb/tradesuite`, `TLHOME=$TSHOME/tradelens` | `echo $TLHOME` |
| Shared venv activated | `source $TSHOME/sourceme.sh` |
| PostgreSQL 16 reachable on host:port from `etc/config.yml` | `pg_isready -h 127.0.0.1 -p 5432` |
| `tradelens` role can connect | `psql -h 127.0.0.1 -p 5432 -U tradelens -d tradelens -c '\dt' \| head` |
| For test-DB bootstrap: role has CREATEDB or superuser | `psql -U tradelens -d postgres -c '\du tradelens'` |

`etc/config.yml` is the single source of truth for connection details. Every
script in this directory reads it via `tradelens.core.config`.

## TL;DR — fresh install order

```bash
source $TSHOME/sourceme.sh
cd $TLHOME

# 1. Create database role + empty database (manual, one-time)
#    See "Bootstrapping the database role" below.

# 2. Create the schema and mark all historical migrations applied
python3 bin/setup/setup_database_pg.py

# 3. Apply any migrations newer than the snapshot (no-op on a clean install)
python3 bin/setup/migrate.py up

# 4. Seed accounts from etc/accounts.yml
python3 bin/setup/sync_accounts.py
```

After this, restart the API (`./bin/start_api.sh`) and the install is ready.
For ongoing operation, only step 3 (`migrate.py up`) and step 4
(`sync_accounts.py` after editing `accounts.yml`) ever need to be re-run.

## Script reference

### `setup_database_pg.py` — schema bootstrap

Creates every TradeLens table with the correct columns, indexes, and
constraints. After creating tables it calls `mark_all_migrations_applied`,
recording a row in `schema_migration` for every `.sql` file in `migrations/`
so subsequent `migrate.py` runs see them as already-applied.

| Aspect | Detail |
|---|---|
| When to run | Once on a fresh database, or when you want to drop and recreate from scratch |
| Idempotent | Yes when run without `--recreate` (skips existing tables); destructive with `--recreate` |
| Modifies prod data | **Yes with `--recreate`** — drops and recreates every table |
| Reads | `etc/config.yml` (`postgresql:` section), `migrations/*.sql` (for checksums) |
| Writes | All tables in `expected_tables` list; `schema_migration` rows |

**Flags**

```text
--recreate              Drop and recreate existing tables (DESTRUCTIVE)
--verify-only           Just verify all expected tables exist; no DDL
--mode {bootstrap,production}
                        Identity column mode. Default 'production' uses
                        GENERATED ALWAYS AS IDENTITY. 'bootstrap' uses
                        GENERATED BY DEFAULT AS IDENTITY which permits
                        explicit IDENTITY_INSERT — only needed during
                        a data-migration ETL.
--debug                 Verbose logging
```

**Common failure modes**

- `No 'postgresql' section in config.yml` — the script reads `config.postgresql`
  (not `config.database`). Both sections exist in the canonical config. If
  yours is missing the `postgresql:` block, copy it from the example.
- `permission denied for schema public` — the configured user needs CREATE
  on schema public. Grant via the maintenance role.

### `migrate.py` — migration runner

The forward-going way to apply schema changes. Tracks applied state in
`schema_migration` (filename, checksum, duration, applied_by). Each migration
runs in its own transaction; a failure rolls that file back and exits
non-zero, leaving subsequent migrations unapplied.

| Aspect | Detail |
|---|---|
| When to run | After every `git pull` that adds files under `migrations/`, before restarting services |
| Idempotent | Yes — already-applied files are skipped on the next run |
| Modifies prod data | Only the changes inside the migration files |
| Reads | `etc/config.yml`, `migrations/*.sql` |
| Writes | Whatever the migration SQL writes; `schema_migration` row per applied file |

**Subcommands**

```text
migrate.py status                   Show applied vs pending migrations + checksum drift
migrate.py up                       Apply all pending migrations in filename order
migrate.py up --dry-run             Show what would apply without running
migrate.py mark-applied <file>      Record a file as applied without running its SQL
                                    (use after a manual hotfix or split migration)
```

All three subcommands accept `--database <name>` to override the database
name from config (host/port/user/password still come from config). Used by
`setup_test_db.py` to operate on `tradelens_test`.

**Behaviour notes**

- `status` prints "Checksum drift" warnings when an applied file's checksum
  differs from disk. This is expected for trivial fixups (typo in a comment)
  but a real edit means the production schema and the file are out of sync.
- `up` warns about drifted files but does not re-apply them; reapply manually
  if the change is real.
- `mark-applied` is the right way to recover if a migration was run by hand
  via psql.

### `run_migration.py` — DEPRECATED shim

Old per-file runner. Now just prints a deprecation notice and exits 2. Kept
for muscle memory / external scripts that hardcoded the old name. Use
`migrate.py up` instead.

### `mark_all_migrations_applied.py` — one-time migration registry backfill

One-shot script for the day `schema_migration` was first introduced.
Iterates every `.sql` file in `migrations/` and inserts an `ON CONFLICT DO
NOTHING` row marking it applied. After a normal fresh install,
`setup_database_pg.py` does this for you (it calls the same logic
internally), so this standalone script is only needed if you somehow have a
populated database that pre-dates the `schema_migration` table.

| Aspect | Detail |
|---|---|
| When to run | One-off: an existing DB that doesn't yet have `schema_migration` populated |
| Idempotent | Yes |
| Modifies prod data | No — only writes to `schema_migration` |

### `sync_accounts.py` — accounts.yml → DB

Reads `etc/accounts.yml`, upserts each account into `accounts`, and marks any
DB-only accounts as inactive. After a successful run it prints "Restart the
API" — required because the API caches account ID mappings at startup.

| Aspect | Detail |
|---|---|
| When to run | After every edit to `etc/accounts.yml` |
| Idempotent | Yes |
| Modifies prod data | Yes — `accounts` table only |
| Side effects | API must be restarted to pick up new IDs |

No CLI flags. Reads `etc/config.yml` for DB credentials and `etc/accounts.yml`
for the account list.

### `setup_test_db.py` — test database bootstrap

Creates the isolated `tradelens_test` database used by `pytest -m
integration`. Reuses `PGDatabaseSetup` from `setup_database_pg.py` so the
test DB is byte-compatible with a fresh production install. Also creates an
empty `market_candle` schema (production created it on-demand from the now-
deleted `mdsync` service) and seeds accounts from `accounts.yml`.

Refuses to run if `--database-name` matches the production database in
config — protects against fat-fingered `--recreate`.

| Aspect | Detail |
|---|---|
| When to run | First-time CI / dev test setup, or when reseeding |
| Idempotent | Yes (without `--recreate`); use `--recreate` to fully rebuild |
| Modifies prod data | **Never** — refuses if target name == production DB name |

**Flags**

```text
--recreate                  Drop and recreate the test DB (DESTRUCTIVE for test data only)
--database-name NAME        Target DB (default: tradelens_test)
--skip-seed                 Don't seed accounts from accounts.yml
--debug                     Verbose logging
```

For day-to-day usage from the project root:

```bash
python3 bin/setup/setup_test_db.py                  # First time
python3 bin/setup/setup_test_db.py --recreate       # Reset between branches
python3 bin/setup/migrate.py up --database tradelens_test   # Apply new migrations
```

Test fixtures (see `tests/conftest.py`) connect to whichever database is
named in `etc/config.yml` `database.test_database` — so the
`--database-name` you pass here must match that.

### `migrate_idea_items_to_trades.py` — one-time data migration

For trades that were created from ideas before the copy-on-creation feature
shipped: copies notes/tags/snapshots from the parent `trade_idea` to the
linked `trade_journal` row. Skips duplicates.

| Aspect | Detail |
|---|---|
| When to run | One-off, already run in production. Safe to re-run (skips dups). |
| Idempotent | Yes — duplicate-detection by content + event_type |
| Modifies prod data | Yes — inserts into `trade_journal_notes` |

```text
--dry-run        Show what would be copied without inserting
```

### `migrate_parser_inbox.py` — one-time data migration

Creates the `parser_inbox` table (idempotent `CREATE TABLE IF NOT EXISTS`)
and backfills the last N days of inbox rows from `trade_journal_notes` +
`trade_idea`. Used once when the Inbox feature shipped so it had history to
display on first open.

| Aspect | Detail |
|---|---|
| When to run | One-off, already run. Safe to re-run with `--skip-backfill` to just verify the table exists. |
| Idempotent | Yes — uses ON CONFLICT on `(source, source_message_id)` |
| Modifies prod data | Yes — writes to `parser_inbox` |

```text
--days N            Backfill window in days (default: 30)
--dry-run           Count without inserting
--skip-backfill     Just create the table + indexes, no backfill
```

Note: this script reads DB credentials from `PG_HOST` / `PG_PORT` /
`PG_DATABASE` / `PG_USER` / `PG_PASSWORD` env vars, **not** from `config.yml`
like the others. If those env vars aren't set, defaults match the canonical
config (`127.0.0.1:5432`, `tradelens`/`tradelens_poc`).

### `backfill_position_idx.py` — one-time data migration

Fills in `position_idx` on legacy `trade_journal` rows that pre-date the
position_idx implementation. Strategy:

1. First, copy `position_idx` from any associated leg via `trade_leg_map` →
   `order_leg_hist`.
2. With `--infer-from-side`, additionally infer for futures trades from
   `trade_journal.side`: long → 1, short → 2 (hedge mode assumption).
3. Then propagate the resolved `position_idx` back down to
   `order_leg_hist` rows so live-leg attachment works.

| Aspect | Detail |
|---|---|
| When to run | One-off after enabling hedge-mode position tracking. Already run in production. |
| Idempotent | Yes — re-run on a fully-backfilled DB is a no-op (no NULL rows to find) |
| Modifies prod data | Yes — `trade_journal.position_idx`, `order_leg_hist.position_idx` |

```text
                        Default is dry-run
--apply                 Apply changes
--infer-from-side       Also infer from side for futures (hedge mode)
--debug                 Verbose logging
```

Run dry-run first, eyeball the output, then re-run with `--apply`.

## Legacy Sybase artifacts (do not run)

The following files date from the pre-March-2026 Sybase ASE installation
and are kept only as reference for the (now-deleted) Sybase server. They
will not work against the current PostgreSQL database.

| File | Was for |
|---|---|
| `compare_databases.sh` | Schema diff `tradelens` vs `tradelens_bkup` via `sqsh` against ASE |
| `remove_identity_gaps.sh` | Compact ASE identity sequences with FK remapping |
| `show_identity_gaps.sh` | Report ASE identity-gap settings |
| `show_identity_gaps.sql` | Same, as a `sqsh` script |
| `set_identity_gap.sql` | `sp_chgattribute` calls per table |
| `set_identity_gap-ALL_TO_ONE_TEMP.sql` | Workspace SQL from the gap-compaction effort |
| `truncate.sql` | `truncate table` chain on ASE |
| `identity_references.conf` | FK reference map consumed by `remove_identity_gaps.sh` — see also `MEMORY.md` warning about overlapping IDs being intentional |

PostgreSQL identity columns (`GENERATED ALWAYS AS IDENTITY`) don't have the
same gap-on-restart problem ASE had with `identity_burning_set_factor`, so
there is no PG equivalent of these scripts and none is needed.

If you need to compact PG identity sequences anyway, **stop first** — see
`MEMORY.md` → "order_leg_live / order_leg_hist: Overlapping IDs Are By
Design". Multiple sessions have broken the application by trying to fix
overlap that is not actually broken.

## Recovery and edge cases

### "I edited a migration after it was applied"

`migrate.py status` prints a checksum drift warning. If the edit was just a
typo / comment fix, leave it. If it was a real DDL change, write a new
forward migration that takes the schema where you want it.

### "I ran a migration manually via psql instead of `migrate.py up`"

```bash
python3 bin/setup/migrate.py mark-applied 0NN_my_migration.sql
```

This records the file as applied without re-running the SQL.

### "Schema reference (`etc/schema.md`) is out of sync"

```bash
cd $TLHOME && python3 bin/tools/dump_schema.py
```

Fully regenerates `schema.md` from `information_schema`.

### "`setup_database_pg.py --recreate` failed halfway"

The DROP-and-CREATE pass is per-table inside one connection but not one
transaction. Re-run the same command — `setup_database_pg.py` will drop
remaining tables and recreate everything cleanly.
